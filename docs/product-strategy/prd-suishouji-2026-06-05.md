## 1. 项目背景与目标

### 1.1 项目背景

中国移动支付市场已高度成熟，微信支付和支付宝覆盖了绝大多数日常消费场景。然而，现有记账 App 普遍存在以下痛点：

- **记账摩擦高**：用户需要手动打开 App、输入金额、选择分类，流程繁琐
- **遗忘率高**：消费后忘记记录，导致账单不完整
- **自动化不足**：虽然有支付通知，但大多数 App 不能自动转化为记账条目
- **隐私担忧**：大量记账 App 强制上传数据到云端，用户对财务隐私敏感

随记账（英文代号 CashFlow / MoneyNote）正是为解决这些痛点而生——利用 Android 系统能力自动捕获支付信息，通过 AI 辅助识别与分类，同时坚持数据本地优先的隐私原则。

### 1.2 产品愿景

让每个中国智能手机用户都能以"零摩擦"的方式完成记账——消费即记录，记录即洞察。

### 1.3 产品目标（3 个清晰、正交的目标）

| # | 目标 | 衡量标准 |
|---|------|---------|
| G1 | **降低记账摩擦**：通过自动识别和 AI 对话，让 80% 的日常消费无需手动录入 | 自动识别覆盖率 ≥ 80%；通知解析准确率 ≥ 85%；OCR 金额准确率 ≥ 80% |
| G2 | **提升消费洞察**：通过图表、预算和趋势分析，帮助用户理解并优化消费结构 | 月活跃用户的图表页访问率 ≥ 60%；预算设置率 ≥ 40% |
| G3 | **保障隐私与数据主权**：数据默认本地存储，云端同步为可选增强 | 本地离线可用率 100%；云端同步成功率 ≥ 99%；零隐私投诉 |

### 1.4 目标设备与使用场景

- **目标设备**：华为 Mate 70 Pro / HarmonyOS 4.x（APK 兼容模式）
- **目标用户**：中国本土个人消费者，日常使用微信/支付宝进行移动支付
- **使用频率**：每日多次（自动触发）+ 每周 2-3 次（主动查看/分析）
- **使用场景**：通勤、午休、周末回顾消费

## 4. 功能需求（详细）

### 4.1 智能记账引擎（核心差异化）

#### 4.1.1 支付通知识别

**描述**：通过 `NotificationListenerService` 监听微信和支付宝的支付通知，自动解析并生成待确认草稿。

**详细规则**：
- 监听微信包名 `com.tencent.mm` 和支付宝包名 `com.eg.android.AlipayGphone`
- 解析通知标题和内容，提取：金额、商户名、交易类型（支出/收入/转账）
- 金额提取使用正则匹配，支持标签金额（"支付金额：¥35.00"）、货币符号金额（"¥35.00"）、元后缀金额（"35元"）
- 商户提取：识别标题中的商户名，匹配"向XX付款"、"XX商户"等模式
- 按包名自动推断账户：微信 → `wechat`，支付宝 → `alipay`，其他 → `cash`
- 通过指纹去重：`notif:<包名>:<时间桶>:<金额>:<商户>:<文本hash>`

**去重策略**：
- 使用"来源包名 + 2 分钟时间桶 + 金额 + 商户 + 文本摘要 SHA256"构建指纹
- 同一指纹的通知不重复生成草稿

**验收标准**：
| # | 验收条件 | 目标值 |
|---|---------|--------|
| A1 | 通知解析后必须生成 DRAFT 状态草稿，不能直接 CONFIRMED | 100% |
| A2 | 金额提取准确率（对比实际支付金额） | ≥ 85% |
| A3 | 商户名提取覆盖率（至少有一个可识别商户名） | ≥ 80% |
| A4 | 同一支付去重率（不产生重复草稿） | ≥ 95% |
| A5 | 未开启通知权限时有引导提示 | 必须 |

#### 4.1.2 OCR 截图识别

**描述**：用户从相册导入微信/支付宝支付截图，或通过截图监听自动捕获，使用华为 ML Kit OCR 引擎识别文字并生成草稿。

**支持的截图类型**：
1. **单笔支付详情页**：提取页面中央金额、商户名、支付时间、交易单号
2. **多笔账单列表页（微信）**：按行拆分成多条草稿
3. **多笔交易记录列表（支付宝）**：支持"昨天/今天"相对时间、"05-17 12:34"紧凑格式

**OCR 后处理规则**：
- 商户标签行（"商户简称/商户全称"）被 OCR 拆分时，跳过标签碎片，提取真实商户名
- 列表页的月度汇总、红包抵扣、积分等辅助字段不导入为交易
- "交易关闭"和正数金额行暂不导入为消费草稿
- 多笔结果识别后全部进入草稿队列

**去重策略**（三级优先级）：
1. 存在交易单号时：使用 `ocr:txn:<交易单号后48位>`
2. 存在支付时间时：使用 `ocr:<金额>:<商户>:<支付时间>`
3. 兜底：使用 `ocr:<金额>:<商户>:<文本SHA256前24位>`

**相册导入特殊规则**：
- 相册导入允许同一图片再次生成草稿（防止用户重试被去重拦截）
- 同一图片的重复导入将复用/更新现有草稿而非创建新草稿
- 与已确认流水重复时保留草稿（给用户判断空间）

**验收标准**：
| # | 验收条件 | 目标值 |
|---|---------|--------|
| B1 | 导入截图后至少生成一条 DRAFT 草稿 | 100% |
| B2 | OCR 失败时草稿仍可见（显示 OCR_FAILED 状态） | 100% |
| B3 | 单笔详情页金额提取准确率 | ≥ 80% |
| B4 | 多笔列表页每笔独立生成草稿 | ≥ 90%（遗漏率 < 10%） |
| B5 | OCR 草稿支持重识别（Retry） | 必须 |

#### 4.1.3 截图监听服务

**描述**：通过前台服务监听系统截图目录的新图片，自动触发 OCR 识别。

**当前状态**：服务框架已搭建（`ScreenshotMonitorService`），但需真机验证。

**关键依赖**：
- Android 媒体存储读写权限
- 前台服务权限
- HarmonyOS 4.x 兼容性验证

**验收标准**：
| # | 验收条件 | 目标值 |
|---|---------|--------|
| C1 | 截图后 30 秒内生成 OCR_PENDING 草稿 | ≥ 80% |
| C2 | 非支付截图不误识别为消费 | 误报率 < 15% |
| C3 | 服务耗电量可接受 | 日均额外耗电 < 3% |

### 4.2 草稿审核系统

#### 4.2.1 草稿生命周期

```
支付通知 / 相册导入 / 截图监听
          │
          ▼
    ┌─────────┐
    │  DRAFT  │  ← 所有自动来源初始状态
    └────┬────┘
         │
    ┌────┼────────────┐
    ▼    ▼            ▼
┌──────┐ ┌────────┐ ┌────────┐
│CONFIRM│ │IGNORED │ │DELETED │
│  ED   │ │        │ │        │
└──┬───┘ └────────┘ └────────┘
   │
   ▼
  正式入账统计
```

**草稿状态机**：
| 当前状态 | 操作 | 目标状态 | 说明 |
|---------|------|---------|------|
| DRAFT | 编辑 | DRAFT | 修改金额/分类/账户/备注 |
| DRAFT | 确认 | CONFIRMED | 进入正式统计 |
| DRAFT | 忽略 | IGNORED | 不进入统计但保留记录 |
| DRAFT | 删除 | DELETED | 物理删除 |
| OCR_FAILED | 重识别 | OCR_PENDING → DRAFT/OCR_FAILED | 重新触发 OCR |
| OCR_FAILED | 删除 | DELETED | 物理删除 |
| CONFIRMED | 编辑 | CONFIRMED | 修改后仍为已确认 |
| CONFIRMED | 删除 | DELETED | 物理删除，从统计中移除 |

**验收标准**：
- CONFIRMED 流水才进入首页/账单/图表的统计数据
- IGNORED 流水不进入统计但保留在账单列表中
- DRAFT 流水在首页和账单页的待确认分区可见
- 所有草稿操作（编辑/确认/忽略/删除）实时生效

### 4.3 手动记账

**描述**：用户主动记录的快捷入口。

**入口**：
- 底部导航中间「+」按钮
- 首页 FAB（Floating Action Button）

**弹窗字段**：
| 字段 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| 金额 | 数字输入 | 是 | 空 | 以元展示，以分存储，自动聚焦 |
| 类型 | 支出/收入切换 | 是 | 支出 | 切换后分类列表跟随变化 |
| 分类 | 网格选择 | 是 | 当前类型第一个分类 | 与设置页分类管理同步 |
| 账户 | 下拉选择 | 是 | 账户列表第一个，兜底现金 | 与设置页账户管理同步 |
| 备注 | 文本输入 | 否 | 空 | 可选备注或商户名 |
| 导入截图 | 按钮 | 否 | — | 点击后关闭弹窗，打开图片选择器 |

**特殊行为**：
- 打开弹窗后自动聚焦金额输入框
- 保存后金额和备注自动清空（方便连续记账）
- 手动记账直接生成 CONFIRMED 状态（跳过草稿审核）
- 分类较多时弹窗可滚动，保存按钮始终可见

**验收标准**：
- 金额为空或非法时不保存
- 金额以元输入、以分存储
- 保存后首页和账单页数据刷新

### 4.4 首页概览

**描述**：用户打开 App 的第一屏，快速回答"本月花了多少、有没有超预算、主要花在哪些分类"。

**页面结构**（从上到下）：

**4.4.1 月度概览卡片**
- 品牌标题 `CashFlow`（左侧）+ 搜索/通知图标（右侧）
- 当前月份（如"2026年6月"）
- 本月支出（大数字，红色/黑色）
- 月预算进度条（绿色 → 黄色 80% → 红色 100%）
- 预算剩余额度（"剩余 ¥1,200.00"）
- 本月收入 + 结余（小字）

**4.4.2 TOP 分类卡片**
- 本月消费 Top 3 分类，三列等宽卡片
- 每张卡片：分类图标 + 名称 + 金额 + 占比 + 分类色进度条
- 空状态："确认流水后会显示本月消费最高的分类"

**4.4.3 待确认草稿入口**
- 草稿数量角标
- 点击进入草稿列表

**4.4.4 近期账单**
- 最近 5 条确认流水
- 显示分类、商户、金额、时间

**验收标准**：
- 本月支出只统计当前月份 CONFIRMED 支出
- 预算为 0 时不显示进度警告
- 预算 80%+ 预警，100%+ 超支
- 无数据时展示友好的空状态引导

### 4.5 账单列表

**描述**：用户查看和管理所有流水的页面。

**页面结构**：

**4.5.1 月历支出卡**
- 月份标题 + 左右切换按钮
- 周日→周六标题行

### 4.10 远程同步系统

**描述**：为已登录用户提供数据备份和多设备同步能力。

**同步策略**：
1. **本地优先**：用户操作写入 Room 后立即返回，不等待网络
2. **增量同步**：基于 `client_id`（UUID）实现增量上传/下载
3. **冲突处理**：以 `client_id` 为唯一标识，后写入覆盖（Last-Write-Wins）
4. **离线可用**：无网络时完全正常使用，连网后自动同步

**API 接口**：
| 方法 | 路径 | 说明 |
|------|------|------|
| POST | /api/sync/upload | 批量上传本地变更 |
| GET | /api/sync/download | 下载服务端变更（since 参数） |

**数据映射**：
| 客户端字段 | 服务端字段 | 说明 |
|-----------|-----------|------|
| `LedgerEntryEntity.id` | `client_id` | UUID，同步冲突检测依据 |
| `amountCents` | `amount_cents` | 金额（分） |
| `categoryNameSnapshot` | `category_name` | 分类名快照 |
| `accountNameSnapshot` | `account_name` | 账户名快照 |

**验收标准**：
- 本地写入后立即可见，无需等待网络
- 同步成功后服务端数据与本地一致
- 网络恢复后自动补齐未同步数据
- 同步失败不影响本地功能

### 4.11 用户认证系统

**Token 管理**：

### 4.12 AI 对话记账（Phase 3）

**描述**：用户通过自然语言对话完成记账、查询和统计分析。

**技术架构**：
- 前端：Jetpack Compose ChatScreen（多会话、消息气泡、输入框）
- 后端：Spring Boot `AiChatService` → DeepSeek API（deepseek-chat）
- 上下文：Redis 存储最近对话历史
- Function Calling：4 个 Tool

**支持的对话场景**：

| 场景 | 示例输入 | 系统行为 |
|------|---------|---------|
| 快速记账 | "帮我记一笔，中午吃饭花了 35 块" | LLM 提取信息 → 调用 add_transaction Tool → 返回确认卡片 |

| 趋势分析 | "看看这个月的消费趋势" | LLM → 调用 get_spending_summary → 返回统计 |
| 多轮补全 | "记一笔打车费" → "27块" → "交通分类" | LLM 逐步补全信息 |

**4 个 Function Calling Tool**：
| Tool | 功能 | 参数 |
|------|------|------|
| `add_transaction` | 新增记账草稿 | content（自然语言描述）, date（可选） |
| `query_transactions` | 查询历史流水 | start_date, end_date, category, type |
| `get_spending_summary` | 消费统计汇总 | start_date, end_date, group_by |
| `get_category_by_name` | 模糊匹配分类名 | name |

**当前状态与待优化**：
- ✅ 后端 AI 服务已实现，Function Calling 正常工作
- ✅ ChatScreen 已实现（多会话、消息气泡）
- ⚠️ 确认卡片与 Tool calls 格式兼容需优化
- ⚠️ 多轮对话补全信息体验需打磨
- 📋 当前参考皮肤中对话 Tab 暂不展示在底部导航

**验收标准**：
- AI 能完成记账、查询、统计 3 类核心场景
- AI 记账结果在 Android 端展示确认卡片（可编辑再确认）
- 自动识别结果先进草稿，不直接入账
- 对话历史可跨会话恢复（从 Redis）

## 9. 开发原则与约束

### 9.1 核心原则（不可妥协）

1. **自动来源永远先进草稿**：任何自动识别的结果必须先进入 DRAFT 状态
2. **用户确认是入账边界**：只有 CONFIRMED 的流水才进入正式统计
3. **金额分存储、元展示**：内部计算使用分（Long），UI 展示使用元
4. **统计只使用 CONFIRMED**：首页、图表、预算计算只统计已确认流水
5. **OCR 失败必须可见**：OCR 失败不静默丢弃，草稿始终可见
6. **本地优先、云端增强**：所有功能离线下可用，同步为可选增强

### 9.2 开发约束

- Room 迁移必须保护已有数据，禁止 destructive migration
- 新功能优先保证本地可用，再考虑云端或同步
- UI 变更保持与现有 Compose 风格一致
- 修改解析/OCR 行为时，同步更新去重逻辑
- 代码集中时逐步拆分（当前 MainActivity.kt 偏重）
